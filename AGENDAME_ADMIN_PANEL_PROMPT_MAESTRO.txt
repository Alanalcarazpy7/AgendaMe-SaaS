AGENDA ME — PROMPT MAESTRO PARA PANEL PRIVADO DEL PROPIETARIO DEL SAAS
=====================================================================

OBJETIVO GENERAL
----------------
Implementar un panel administrativo privado, profesional y completo para el propietario de AgendaMe, separado del dashboard de los negocios.

Este panel debe permitir administrar toda la plataforma SaaS sin depender de entrar constantemente a Supabase o ejecutar SQL manual.

El objetivo principal es la funcionalidad, seguridad y control del negocio. El diseño debe ser profesional, claro y coherente con AgendaMe, pero no es la prioridad absoluta en esta fase.

IMPORTANTE:
- No te limites a analizar o recomendar.
- Inspecciona el proyecto real e implementa directamente.
- Reutiliza lo que ya existe.
- No inventes datos.
- No sustituyas datos reales por mocks permanentes.
- No afirmes que algo funciona si no fue verificado.
- No marques migraciones como aplicadas si solo se creó el archivo.
- No ejecutes cambios remotos en Supabase sin autorización explícita.
- Mantén npm run build funcionando.
- Mantén compatibilidad con la lógica actual.
- No elimines ni rompas funcionalidades existentes.
- Trabaja por fases y documenta el progreso para que otro agente pueda continuar.

=====================================================================
SEGURIDAD CRÍTICA Y NO NEGOCIABLE
=====================================================================

La seguridad no debe depender de ocultar la ruta.

1. Identidad del propietario

Usar una variable de entorno exclusivamente de servidor:

ADMIN_OWNER_USER_ID=<UUID de auth.users del propietario>

Agregarla a:
- .env.example
- documentación

Nunca escribir el UUID real ni secretos en Git.

Reglas:
- Si ADMIN_OWNER_USER_ID no existe, el panel debe fallar cerrado.
- El usuario debe estar autenticado.
- El UUID debe coincidir exactamente con ADMIN_OWNER_USER_ID.
- El perfil debe tener rol_global = 'super_admin'.
- Debe cumplir ambas condiciones.
- No usar email como identificador principal.

2. Guard central

Crear una función única y reutilizable, por ejemplo:

requirePlatformOwner()

Debe:
- ejecutarse solo en servidor;
- obtener el usuario autenticado real;
- verificar ADMIN_OWNER_USER_ID;
- verificar perfiles_usuario.rol_global = super_admin;
- fallar de forma segura;
- devolver solo datos mínimos del propietario autorizado.

También puede existir:

getPlatformOwnerOrNull()

3. Protección del layout

Proteger:
- src/app/admin/layout.tsx

Reglas:
- no renderizar datos antes de autorizar;
- visitante no autenticado: redirección segura al login;
- usuario autenticado no autorizado: notFound() o respuesta segura;
- no revelar información interna del panel.

4. Protección de acciones y APIs

Cada operación administrativa debe volver a ejecutar requirePlatformOwner():
- Server Actions;
- Route Handlers;
- exportaciones;
- descargas;
- mutaciones;
- consultas privilegiadas;
- aprobación de pagos;
- cambios de plan;
- bloqueos;
- desbloqueos;
- modificaciones de usuarios;
- cambios de precios.

Nunca confiar solamente en que la llamada viene desde /admin.

5. Supabase privilegiado

- Reutiliza el cliente administrativo server-only si ya existe.
- Nunca importar service role en Client Components.
- Nunca usar NEXT_PUBLIC para secretos.
- Nunca devolver secretos en props o respuestas.
- Mantener módulos privilegiados con server-only.
- No crear un segundo sistema de autenticación.

6. Base de datos

Las nuevas RPC administrativas deben:
- validar es_super_admin();
- usar security definer solo cuando corresponda;
- usar set search_path seguro;
- revocar execute de PUBLIC si son sensibles;
- registrar auditoría;
- validar parámetros;
- mantener compatibilidad.

7. Caché

Todo /admin debe ser privado y dinámico:
- no-store o estrategia equivalente;
- no cachear públicamente;
- no generar estáticamente;
- no usar CDN público para respuestas administrativas.

8. Validación

- Validar entradas con Zod en servidor.
- Confirmar operaciones sensibles.
- Evitar doble envío.
- Deshabilitar botones durante mutaciones.
- No mostrar stack traces.
- No culpar al usuario en errores.
- Mostrar mensajes claros y recuperables.

9. Auditoría

Registrar:
- cambio de plan;
- cambio de vencimiento;
- activación;
- cancelación;
- bloqueo;
- desbloqueo;
- aprobación de pago;
- rechazo de pago;
- edición de plan;
- cambio de precios;
- exportaciones sensibles;
- acciones sobre usuarios;
- cambios administrativos relevantes.

Guardar:
- administrador;
- negocio afectado;
- acción;
- tabla;
- registro;
- fecha;
- origen;
- detalles anteriores y nuevos cuando sea razonable.

No guardar:
- contraseñas;
- tokens;
- claves;
- secretos.

10. MFA

Si el proyecto ya soporta MFA/AAL2:
- exigir AAL2 para /admin.

Si no:
- no inventar un flujo falso;
- preparar variable opcional:

ADMIN_REQUIRE_AAL2=true|false

- documentar cómo habilitarlo más adelante.

11. Reglas adicionales

- No CORS abierto.
- No open redirects.
- No usar localStorage para permisos.
- No exponer auth.users al navegador.
- No mostrar tokens de invitación.
- No registrar secretos en consola.
- No aceptar IDs sin validar existencia.
- No confiar en roles enviados por el cliente.
